### 8. 任务状态通知机制

每个工作流运行对应一个独立会话窗口，实时推送节点状态变更：

**通知时机**：
- 节点状态变化（running → waiting_user_input → completed/failed）
- 节点进入 waiting_user_input 时，触发前端交互模式
- 运行整体完成或失败
- 运行异常需要人工介入

**通知方式**：
- WebSocket 推送至对应的任务会话窗口
- 飞书消息卡片（可点击跳转至前端交互页面）

**会话隔离**：每个 run_id 对应一个独立的 WebSocket 连接和通知窗口，用户可以同时监控多个运行任务。

### 9. 常驻 Agent 对话窗口

常驻一个独立的 Agent 对话窗口，不属于任何具体任务：

**职责**：
- 接收用户随时输入的临时指令（如"帮我查一下XX仓库的最近提交"）
- 跨任务上下文：可以同时了解多个任务的进展
- 作为中央协调入口：用户从一个地方发起所有与 SHRIMP/CC Lead 的交互

**与任务窗口的区别**：

| | 任务状态通知窗口 | 常驻 Agent 窗口 |
|---|---|---|
| 作用域 | 单个 run_id | 全局/跨任务 |
| 生命周期 | 运行开始创建，运行结束关闭 | 常驻不关闭 |
| 内容 | 节点状态变更通知 | 自由对话 + 任务调度 |
| 典型消息 | "节点 code_review 已进入等待确认状态" | "帮我看看当前所有运行的状态" |

**技术实现**：
- 常驻窗口维护一个独立的 WebSocket 连接（不同于任务窗口）
- pi-agent 在常驻窗口中扮演中央协调角色，可访问所有任务上下文
- 通过后端 API 查询所有 run_id 的状态，不限于单一任务
